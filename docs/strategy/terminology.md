# 术语表

本文件定义 `spec-toolchain` / `spec-alchemy` 的严格术语规范，用于统一产品、设计、工程、测试与 agent 文档中的表达，避免因同义替换、临时命名或中英混排失控而造成战略与实现漂移。

适用范围：

- `docs/strategy/` 下的全部战略文档
- 面向产品线定义的设计文档、提案文档、评审文档
- 新增 `xxx-spec` family 的命名、说明、CLI 帮助、viewer 文案、agent 指南

使用原则：

- 优先使用“规定写法”，不要自由替换。
- 如无必要，不新增同义词。
- 必须保留英文的术语，统一按本表写法使用。
- 中文叙述为主，术语按本表决定是否保留英文。

## 1. 品牌与产品线术语

### `Spec Toolchain`

- 规定写法：`Spec Toolchain`
- 中文说明：产品线工作区名称
- 适用语境：仓库、工作区、产品线总称
- 禁止写法：
  - `spec toolchain`
  - `SpecToolchain`
  - `工具链平台`
- 说明：作为专有名词时保持大小写，不翻译。

### `spec-alchemy`

- 规定写法：`spec-alchemy`
- 中文说明：品牌/组织标识
- 适用语境：品牌、workspace、npm scope、仓库归属
- 禁止写法：
  - `Spec Alchemy`
  - `spec alchemy`
  - `规格炼金术`
- 说明：在代码仓库与产品语境中统一保留连字符写法。

### `AI-native spec tooling`

- 规定写法：`AI-native spec tooling`
- 中文说明：AI 原生 spec 工具体系
- 适用语境：品类定义、品牌定位
- 禁止写法：
  - `AI spec tools`
  - `AI 规范工具`
  - `AI 文档工具`
- 说明：这是品类定义，不要弱化成泛化的 AI 工具表述。

### `spec as code`

- 规定写法：`spec as code`
- 中文说明：将 spec 作为可版本化、机器可读、可执行资产的工作方式
- 适用语境：方法论、产品原则、能力定义
- 禁止写法：
  - `spec-as-code`
  - `规范即代码`
  - `文档即代码`
- 说明：作为核心方法论时保留英文原写法，不替换为相近概念。

## 2. 产品结构术语

### `spec family`

- 规定写法：`spec family`
- 中文说明：一类稳定设计域的 spec 产品族
- 适用语境：产品结构、产品线扩张、准入评审
- 禁止写法：
  - `family`
  - `spec 类型`
  - `spec 模块`
  - `垂类`
- 说明：除非上下文已经非常明确，否则不要省略 `spec`。

### `shared kernel`

- 规定写法：`shared kernel`
- 中文说明：跨 family 复用的共享能力底座
- 适用语境：平台架构、能力边界、产品结构
- 禁止写法：
  - `内核`
  - `公共层`
  - `平台底座`
  - `统一引擎`
- 说明：引用产品结构时统一使用 `shared kernel`，避免出现多个近义说法。

### `wedge product`

- 规定写法：`wedge product`
- 中文说明：楔子产品
- 适用语境：品牌战略、切入点分析
- 禁止写法：
  - `切入产品`
  - `首发产品`
  - `突破口产品`
- 说明：文档中首次出现时可写“`wedge product`（楔子产品）”，之后统一用 `wedge product` 或“楔子产品”，不要混出第三种说法。

## 3. 资产与契约术语

### `source asset`

- 规定写法：`source asset`
- 中文说明：源资产
- 适用语境：输入资产、设计资产、用户自有资产
- 禁止写法：
  - `源文件`
  - `原始文档`
  - `输入材料`
- 说明：强调它是产品资产，而不仅是文件。

### `source of truth`

- 规定写法：`source of truth`
- 中文说明：冲突时的最终判定来源
- 适用语境：评审、建模边界、生成逻辑
- 禁止写法：
  - `真相源`
  - `唯一事实源`
  - `权威来源`
- 说明：固定保留英文，不自行翻译。

### `artifact`

- 规定写法：`artifact`
- 中文说明：由 spec 派生出的可消费产物
- 适用语境：生成物、viewer 输入、分析输出、测试输入
- 禁止写法：
  - `产出物`
  - `衍生文件`
  - `输出件`
- 说明：泛指工具链产生的结构化结果时，统一用 `artifact`。

### `contract`

- 规定写法：`contract`
- 中文说明：稳定约束与可消费边界
- 适用语境：schema、viewer、接口、CLI、生成输出
- 禁止写法：
  - `协议`
  - `约定`
  - `接口规范`
- 说明：当强调稳定边界时统一用 `contract`，不要与网络协议概念混淆。

## 4. 运行时能力术语

### `validation`

- 规定写法：`validation`
- 中文说明：校验总称
- 适用语境：校验体系总称
- 禁止写法：
  - `检查`
  - `验证流程`
- 说明：当不区分 schema 与 semantic 时，用 `validation` 作总称。

### `schema validation`

- 规定写法：`schema validation`
- 中文说明：单文件结构校验
- 适用语境：字段、类型、枚举、格式等约束
- 禁止写法：
  - `结构校验`
  - `Schema 校验`
  - `格式验证`
- 说明：在正式文档中统一用全称。

### `semantic validation`

- 规定写法：`semantic validation`
- 中文说明：跨文件、跨实体、拓扑与所有权校验
- 适用语境：引用关系、边界规则、拓扑规则
- 禁止写法：
  - `语义校验`
  - `语义验证`
  - `业务校验`
- 说明：在正式文档中统一用全称，避免“业务校验”这种过窄表述。

### `analysis`

- 规定写法：`analysis`
- 中文说明：从 spec 推导出的结构化分析结果
- 适用语境：诊断、覆盖、风险、拓扑、影响面
- 禁止写法：
  - `分析层`
  - `分析结果集`
  - `智能分析`
- 说明：作为 runtime surface 之一时统一用 `analysis`。

### `generation`

- 规定写法：`generation`
- 中文说明：基于 spec 生成可消费产物
- 适用语境：代码、配置、测试骨架、viewer artifact
- 禁止写法：
  - `生成层`
  - `代码生成`
  - `导出`
- 说明：当泛指所有生成能力时统一用 `generation`。

### `viewer`

- 规定写法：`viewer`
- 中文说明：面向 inspection 的结构化浏览界面
- 适用语境：产品表面、artifact 消费端、可视化检查
- 禁止写法：
  - `可视化页面`
  - `展示页`
  - `查看器`
- 说明：在本产品语境中统一保留 `viewer`。

### `execution`

- 规定写法：`execution`
- 中文说明：可运行验证或可运行表面
- 适用语境：测试执行、门禁执行、规则运行
- 禁止写法：
  - `执行层`
  - `运行模块`
- 说明：仅在明确指代 runtime surface 时使用。

## 5. 关系与可追溯术语

### `traceability`

- 规定写法：`traceability`
- 中文说明：跨 spec、跨 artifact 的可追溯性
- 适用语境：引用关系、证据链、变更影响、覆盖闭环
- 禁止写法：
  - `追踪性`
  - `可跟踪性`
  - `链路追踪`
- 说明：固定保留英文，避免与 observability 语境混淆。

### `evidence chain`

- 规定写法：`evidence chain`
- 中文说明：证据链
- 适用语境：QA 覆盖、跨 family 引用、设计到验证闭环
- 禁止写法：
  - `证据流`
  - `证据关系`
  - `验证链`
- 说明：当强调“从 spec 到验证结果”的链条时使用。

### `stable ID`

- 规定写法：`stable ID`
- 中文说明：稳定标识符
- 适用语境：跨文件引用、跨 family 引用、可追溯锚点
- 禁止写法：
  - `唯一 ID`
  - `固定 ID`
  - `业务主键`
- 说明：只有强调在长期演进中仍可稳定引用时才使用该术语。

## 6. 文档写作约束

- 中文叙述优先，但术语按本表保留英文，不自行翻译。
- 同一文档内，术语首次出现后不得改用别名。
- 标题可以是中文主标题，术语作为正文或括注出现。
- 面向外部产品定义时，不要把 `ddd-spec` 写成品牌本体；应写为第一个 `spec family` 或当前 `wedge product`。
- 不要把 `spec family` 写成“模块”“模板”“垂类工具”。
- 不要把 `viewer` 写成营销展示页；它是 inspection surface。

## 7. 变更规则

- 若要新增术语，必须先说明它不能复用现有术语的原因。
- 若要替换规定写法，必须同步更新 `docs/strategy/` 下的相关文档。
- 若战略文档、提案文档、实现文档出现术语冲突，以本文件为准，除非本文件先被修订。
