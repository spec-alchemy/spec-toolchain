# Spec Family 准入标准

本文件用于防止 `spec-toolchain` 在新增 `xxx-spec` 时发生战略漂移：只有能成为“可自检、可自省、可执行的 spec as code 设计资产”的 `spec family` 才能进入产品线。

## 定义

`spec family` 不是一组模板或一套写法，而是一类稳定设计域的**可版本化契约**，并且具备可运行的工具链能力（校验、分析、生成、可视化、可执行）。

## 准入门槛（必须同时满足）

1. 设计域成立
   - 对应一个可重复出现的设计域（如产品逻辑、UI、前端架构、QA、架构治理等），不是一次性项目文档。
   - “概念词”必须能落到明确对象：实体/状态/流程/约束/关系/产物。

2. 资产可作为 `source of truth`
   - 有明确的 canonical 输入（machine-readable、diff-friendly、支持 code review）。
   - 能回答“发生争议时以什么为准”：spec 优先于 prose；可生成、可推导部分属于衍生产物。

3. 契约可版本化、可扩展、可收敛
   - 有 schema 版本策略（新增字段/资源种类/引用关系时的兼容原则）。
   - 资源种类（kinds）数量受控，避免“什么都塞进一个万能对象”。
   - 引用（references）有明确的命名与解析规则，跨文件/跨目录可定位。

4. 校验是第一等公民
   - 至少具备：schema validation + semantic validation（跨文件/拓扑/所有权/引用约束）。
   - 校验失败时必须可诊断：错误信息能定位到文件/字段/引用路径，并提供可行动修复方向。
   - 不能把“AI 能理解”当作契约；必须把规则落到可执行校验中。

5. 至少 2 个运行时表面（runtime surfaces）
   - 新 `spec family` 必须提供并证明至少 2 个表面成立：
     - `analysis`：可稳定地产出可被消费的结构化分析结果（不是日志/自然语言总结）。
     - `generation`：可生成面向消费端的产物（代码/配置/测试骨架/文档结构等），可重复构建。
     - `viewer`：可视化或可浏览的结构化呈现（用于 inspection，而不是营销落地页）。
     - `execution`：可执行验证或运行（例如把 QA 断言落地成可运行测试，或把约束落地成门禁）。

6. 可回归、可维护
   - 必须有 repo 内可运行的回归入口：示例输入 + 产物快照/断言 + 测试。
   - 示例必须是“维护者资产”（放在 `examples/` 或等价的 maintainer fixtures），禁止引入业务产品专属资产。
   - 任何影响 `contract` 的改动（schema/kinds/refs/rules）必须同步更新 `semantic validation` 与回归覆盖。

7. 产品边界与发布策略明确
   - 该 `spec family` 的公开边界必须清楚：是否进入公开 CLI 或包边界；命令面与配置面是否稳定。
   - 必须声明成熟度：`experimental` / `beta` / `stable`，并对应不同的兼容承诺与发布门槛。

## 需要的证据（提交评审前必须具备）

1. 最小可用输入（minimum canonical input）
   - 一个可被 code review 的最小 spec 集合（不依赖“外部系统状态”才能解释）。
2. 端到端演示（E2E）
   - 用仓库标准流程完成：校验 ->（分析/生成/可视化/执行中的至少两项）-> 可复现输出。
3. 失败用例
   - 至少 3 个代表性错误：引用缺失、拓扑/所有权违规、跨文件不一致；并验证诊断质量。
4. 回归与门禁
   - 对关键 `contract` 点建立自动化测试；并把新增 `spec family` 纳入持续回归范围。

## 不予准入（典型反例）

- 仅是 prompt 模板、写作规范、Markdown PRD 生成器，没有机器可执行契约与校验。
- 只有 schema，没有 `semantic validation`（跨文件引用、所有权、拓扑约束全靠人或靠 AI 猜）。
- 生成物不可复现（强依赖时序/随机/外部服务），或无法在 repo 中回归验证。
- 资源种类无限膨胀，缺少最小核心模型与演进策略。
- 需要引入业务产品专属数据/界面/素材才能成立（应放在 consumer repo，而不是本仓库）。

## 评审清单

- 该设计域是否稳定、可重复、且能被建模为明确对象与关系？
- canonical 输入是什么？冲突时的 `source of truth` 是什么？
- schema 版本策略与兼容原则是否明确？kinds 是否受控？
- semantic validation 覆盖了哪些跨文件规则？诊断是否可定位/可行动？
- 至少两个 runtime surface 已经实现并能 E2E 复现吗？输出是否结构化且可消费？
- 是否提供了维护者示例与回归测试？是否覆盖失败用例？
- 是否明确公开边界与成熟度（experimental/beta/stable）？命令/配置是否符合整体产品线一致性？
